\# ТЕХНИЧЕСКОЕ ЗАДАНИЕ ДЛЯ РАЗРАБОТКИ: TELEGRAM MINI APP «MORLANA: ИИ-ТАРОЛОГ»



\## ТЕХНИЧЕСКИЙ СТЕК И АРХИТЕКТУРА

\* \*\*Frontend:\*\* React / Vue / Next.js (SPA) + Telegram Web Apps SDK.

\* \*\*Backend:\*\* FastAPI (Python), асинхронный.

\* \*\*База данных:\*\* PostgreSQL (реляционная).

\* \*\*Интеграция с ИИ:\*\* LLM API (GPT-4o-mini для фри-тира, Claude 3.5 Sonnet / GPT-4o для премиума).

\* \*\*Платежи:\*\* Telegram Payments API (ЮKassa / Robokassa).

\## СХЕМА БАЗЫ ДАННЫХ (POSTGRESQL)

```sql

CREATE TABLE users (

&#x20;   telegram\_id BIGINT PRIMARY KEY,

&#x20;   username VARCHAR(255),

&#x20;   real\_name VARCHAR(255) NOT NULL,

&#x20;   birth\_date DATE NOT NULL,

&#x20;   birth\_time TIME NULL, -- NULL означает, что точное время рождения неизвестно

&#x20;   zodiac\_sign VARCHAR(50),

&#x20;   free\_requests\_left INT DEFAULT 3,

&#x20;   subscription\_ends\_at TIMESTAMP WITH TIME ZONE NULL,

&#x20;   created\_at TIMESTAMP WITH TIME ZONE DEFAULT NOW()

);



CREATE TABLE tarot\_sessions (

&#x20;   id UUID PRIMARY KEY DEFAULT gen\_random\_uuid(),

&#x20;   user\_id BIGINT REFERENCES users(telegram\_id) ON DELETE CASCADE,

&#x20;   status VARCHAR(50) DEFAULT 'active', -- 'active' или 'archived'

&#x20;   created\_at TIMESTAMP WITH TIME ZONE DEFAULT NOW()

);



CREATE TABLE chat\_histories (

&#x20;   id BIGSERIAL PRIMARY KEY,

&#x20;   session\_id UUID REFERENCES tarot\_sessions(id) ON DELETE CASCADE,

&#x20;   role VARCHAR(50) NOT NULL, -- 'user' или 'assistant'

&#x20;   content TEXT NOT NULL,

&#x20;   created\_at TIMESTAMP WITH TIME ZONE DEFAULT NOW()

);

```

\## ПОШАГОВЫЙ СЦЕНАРИЙ РАБОТЫ И ЭНДПОИНТЫ



\### Пункт 1. Авторизация и Астро-онбординг (Лид-магнит)

\* \*\*Эндпоинт:\*\* `POST /api/v1/astrology/bonus`

\* \*\*Входящие данные:\*\* `initData` (строка от Mini App), `real\_name`, `birth\_date`, `birth\_time` (может быть NULL).

\* \*\*Логика работы:\*\*

&#x20;   1. Проверить валидность криптографической подписи `initData` через токен бота.

&#x20;   2. Вычислить Знак Зодиака на бэкенде встроенным алгоритмом сопоставления дня/месяца.

&#x20;   3. Создать или обновить юзера в таблице `users` (выдать `free\_requests\_left = 3`, если это новый аккаунт).

&#x20;   4. Отправить запрос в быструю модель (GPT-4o-mini) для генерации персонализированного эмпатичного приветствия-анализа (на 3 абзаца).

&#x20;   5. Вернуть готовый текст астро-разбора на фронтенд. Данный шаг полностью бесплатен для юзера.



\### Пункт 2. Контроль доступа и Paywall (Проверка прав)

\* \*\*Эндпоинт:\*\* `GET /api/v1/tarot/check-access`

\* \*\*Логика работы:\*\* Перед отправкой вопроса пользователя бэкенд проверяет условие: `subscription\_ends\_at > NOW() OR free\_requests\_left > 0`.

\* \*\*Ответ сервера:\*\* Если оба условия ложны, сервер возвращает статус `402 Payment Required`. Фронтенд обязан перехватить этот статус и мгновенно переключить интерфейс на экран оплаты подписки (Paywall).



\### Пункт 3. Механика интерактивного расклада Таро

\* \*\*Эндпоинт:\*\* `POST /api/v1/tarot/draw`

\* \*\*Входящие данные:\*\* `layout\_type` (строка, например "3\_cards"), `count` (целое число, например 3).

\* \*\*Логика работы:\*\* Бэкенд генерирует массив из случайных уникальных чисел (от 1 до 78), соответствующих колоде Райдера-Уэйта.

\* \*\*Безопасность:\*\* Выбор карт происходит \*строго на бэкенде\* (RNG), чтобы исключить подмену карт на клиенте. Массив возвращается фронтенду для рендеринга анимации переворота карт рубашкой вверх.



\### Пункт 4. Трактовка расклада ИИ и Стриминг текста

\* \*\*Эндпоинт:\*\* `POST /api/v1/tarot/predict/stream`

\* \*\*Логика работы:\*\*

&#x20;   1. Если у пользователя нет активной подписки, списать 1 пункт из `free\_requests\_left`.

&#x20;   2. Извлечь из таблицы `chat\_histories` до 4 последних сообщений диалога, привязанных \*только к текущей активной сессии\*.

&#x20;   3. Сформировать системный промпт: Роль (Эмпатичный Таролог) + Профиль (Имя, Знак, Время рождения) + Выпавшие Карты + Текущий вопрос.

&#x20;   4. \*\*Маршрутизация моделей:\*\* Пользователи на триале обслуживаются моделью \*\*GPT-4o-mini\*\* (короткие базовые ответы). Платные подписчики идут на тяжелую премиальную модель (\*\*Claude 3.5 Sonnet / GPT-4o\*\*) для глубокого разбора.

&#x20;   5. Передать текст на фронтенд методом потокового вывода сообщений — \*\*Server-Sent Events (SSE)\*\*.



\### Пункт 5. Управление контекстом сессий

\* \*\*Эндпоинт:\*\* `POST /api/v1/sessions/close`

\* \*\*Логика работы:\*\* Переводит статус текущей сессии в таблице `tarot\_sessions` в состояние `archived`.

\* \*\*Применение:\*\* Вызывается при клике на кнопку «Новый расклад» на фронтенде. Изолирует старые расклады от новых вопросов, предотвращая смешивание контекста ИИ и экономя токены при отправке истории.

\## ЖЕСТКИЕ ПРАВИЛА РАЗРАБОТКИ ДЛЯ НЕЙРОСЕТИ

1\. \*\*Валидация TMA:\*\* Никаких действий в БД на основе голого `telegram\_id`, присланного с клиента. Каждый запрос обязан проходить HMAC-валидацию строки `initData` с использованием `TELEGRAM\_BOT\_TOKEN`.

2\. \*\*Гибкость раскладов:\*\* Архитектура эндпоинтов `/draw` и `/predict` не должна быть завязана жестко на 3 карты. Код должен динамически принимать параметры `count` и `layout\_type`, чтобы в будущем легко переключаться на расклады из 1 или 10 карт.

3\. \*\*Обработка NULL времени:\*\* Если в базе данных `birth\_time` равен `NULL`, системный промпт ИИ должен динамически меняться: \*"Точное время рождения неизвестно. Проведи анализ на основе космограммы, полностью игнорируя сетку астрологических домов"\*.

4\. \*\*Модульность:\*\* Писать чистый, разделенный на слои код (роуты, сервисы ИИ, модели БД). Избегать монолитных файлов.

5\. \*\*Хирургические правки:\*\* Не менять глобально то, где код уже работает. Править только целевые участки, не рефакторить окружение заодно. Минимизировать blast radius изменений.

## Архитектура циклического расклада (1 карта)

### Конечный автомат состояний

```
ОЖИДАНИЕ → ВОПРОС ЗАДАН → КАРТА ВЫТЯНУТА → ИНТЕРПРЕТАЦИЯ → ГОТОВО → ОЖИДАНИЕ (новый цикл)
                                                                    ↘ ЗАВЕРШЕНО (синтез)
```

### Один цикл — последовательность шагов

1. **Пользователь вводит вопрос** → фронтенд отправляет `POST /api/v1/tarot/reading/ask` с `session_id` и `question` → состояние: ВОПРОС ЗАДАН
2. **Пользователь нажимает "Вытянуть карту"** → `POST /api/v1/tarot/reading/draw` → сервер генерирует 1 карту (1-78, random.sample) → состояние: КАРТА ВЫТЯНУТА → показывается анимация переворота
3. **Пользователь нажимает "Получить толкование"** → `POST /api/v1/tarot/reading/interpret` (SSE стриминг) → ИИ интерпретирует карту в контексте вопроса и предыдущих циклов → состояние: ИНТЕРПРЕТАЦИЯ → ГОТОВО
4. **Пользователь выбирает действие:**
   - "Следующий цикл" → `POST /api/v1/tarot/reading/next` → состояние: ОЖИДАНИЕ → переход к шагу 1
   - "Синтезировать расклад" → `POST /api/v1/tarot/reading/synthesis` (SSE) → состояние: ЗАВЕРШЕНО

### Максимум циклов

6 циклов. После 6-го — автоматический переход к синтезу.

### Хранение данных

- **Redis** — активная сессия (состояние, текущий вопрос, текущая карта, история циклов). TTL 1 час.
- **PostgreSQL** — `tarot_sessions` (id, user_id, cycle_count, status) + `reading_cycles` (session_id, cycle_number, question, card_id, interpretation). Записывается при завершении каждого цикла и при синтезе.

### API эндпоинты

| Путь | Метод | Назначение |
|------|-------|------------|
| /api/v1/tarot/reading/start | POST | Создать сессию, вернуть session_id |
| /api/v1/tarot/reading/ask | POST | Принять вопрос, состояние → ВОПРОС ЗАДАН |
| /api/v1/tarot/reading/draw | POST | Вытянуть 1 карту (серверная RNG), состояние → КАРТА ВЫТЯНУТА |
| /api/v1/tarot/reading/interpret | POST | SSE стрим интерпретации ИИ, состояние → ИНТЕРПРЕТАЦИЯ → ГОТОВО |
| /api/v1/tarot/reading/next | POST | Сбросить для нового цикла, состояние → ОЖИДАНИЕ |
| /api/v1/tarot/reading/synthesis | POST | SSE стрим итогового синтеза, состояние → ЗАВЕРШЕНО |
| /api/v1/tarot/reading/state | GET | Текущее состояние сессии (для восстановления после перезагрузки) |

### Промпт ИИ

Системный промпт для интерпретации включает: роль таролога, профиль клиента (имя, знак зодиака, время рождения), историю предыдущих циклов (вопрос, карта, интерпретация), текущий вопрос и текущую карту. Инструкция — дать глубокую интерпретацию, связав с предыдущими циклами.

Промпт для синтеза: список всех циклов → общий вывод и рекомендации.

### Важные замечания

- Rate limiting (slowapi) НЕ применяется к reading-эндпоинтам — авторизация initData đủ для защиты
- В DEV_MODE HMAC-валидация пропускается, заголовок Authorization не обязателен
- Карты хранятся серверно в `reading_cycles` — клиент не может подменить
- initData передаётся в заголовке Authorization (Bearer token), не в query-параметре
